# 1.6.5

## Improvements

### `oauth_mount_prefix`: mounting under a Rack `SCRIPT_NAME`

A new `oauth_mount_prefix` auth value method decouples the path prefix used for
**URL generation** and **`request.path` comparison** from the prefix used for
**route matching**.

> A diagram of the mismatch and the fix is in
> [`doc/mount_prefix_fix.svg`](../mount_prefix_fix.svg).

This targets authorization servers mounted under a sub-path via a Rack
`SCRIPT_NAME` — e.g. `Rack::URLMap.new("/auth" => MyApp)`. In that setup the
mount point is stripped from `PATH_INFO`, so Rodauth route matching keeps
working off `remaining_path` (`/token`) with `prefix` left empty. But
`base_url` and `route_path` do not include `SCRIPT_NAME`, so previously:

* discovery metadata (`oauth_server_metadata_body` /
  `openid_configuration_body`) emitted endpoint URLs without the mount point
  (`https://host/token` instead of `https://host/auth/token`);
* the `issuer` (and `oauth_jwt_issuer`, via `authorization_server_url`) dropped
  the mount point;
* the per-feature `check_csrf?` exemptions for the programmatic endpoints
  (`/token`, `/userinfo`, `/revoke`, ...) compared the full `request.path`
  (`/auth/token`) against the unprefixed `*_path` (`/token`) and missed, so CSRF
  was wrongly enforced on those endpoints.

Set `oauth_mount_prefix` to the mount path to fix all three at once, without
setting Rodauth's global `prefix` (which would prefix *every* feature's routes):

```ruby
rodauth do
  enable :oauth_authorization_code_grant, :oidc
  oauth_mount_prefix "/auth"
  # or derive it from the request:
  # oauth_mount_prefix { request.script_name }
end
```

It defaults to `""`, so behavior is unchanged for servers mounted at the root or
that set `prefix`. It applies *on top of* `prefix` — if `prefix` already accounts
for the mount point (the server is reachable at `remaining_path` including the
prefix), leave `oauth_mount_prefix` empty to avoid doubling it.

#### Contributor note: keeping the mount prefix correct across changes

Endpoint URL helpers (`*_path` / `*_url`) for OAuth routes are regenerated in
`Rodauth::OAuth::FeatureExtensions#auth_server_route` so they honor
`oauth_mount_prefix`. Any route defined via `auth_server_route` gets this for
free. When touching this area, check:

* `grep -rn 'case request.path' lib/` — a new per-feature `check_csrf?` override
  must compare against the route's `*_path` (which is mount-aware), not a hand
  built path.
* New discovery-metadata endpoint fields must be populated from a route's
  `*_url` helper (e.g. `token_url`), not from `"#{base_url}/..."` directly, or
  they will miss the mount prefix.
* `registration_client_uri` (dynamic client registration) is built from
  `base_url` + a route segment rather than a `*_url` helper, so it prepends
  `oauth_mount_prefix` explicitly (in both `oauth_dynamic_client_registration`
  and `oidc_dynamic_client_registration`); keep that in sync if the
  construction changes.
* The management features (`oauth_application_management` /
  `oauth_grant_management`) register their routes via `request.on(<route>)`
  rather than `auth_server_route`, and hand-roll `oauth_applications_path` /
  `oauth_grants_path` from `route_path`. Those helpers feed the management
  views' form actions, `csrf_tag` targets, links, and the post-revoke redirect,
  so they prepend `oauth_mount_prefix` explicitly too; keep that in sync if the
  construction changes.

`oauth_token_revocation`'s `check_csrf?` deliberately keeps CSRF enabled for
form-encoded `/revoke` requests (`!json_request?`); that is independent of the
mount prefix and should not be removed when reasoning about CSRF exemptions.
